# 开发态与浏览态说明

这份文档用于快速区分本项目的源码开发态与 `dist` 浏览态，避免在静态页面开发时反复困惑“到底该改哪里”“为什么页面没生效”。

## 一句话结论

- 开发时优先修改源码目录，不要把 `dist/*.html` 当主战场。
- 查看源码联调效果时，优先打开根目录的 `welcome.html`。
- 查看最终发布结果时，再执行 `npm run build` 后查看 `dist`。

## 两种页面形态

### 开发态

开发态主要由这些目录和文件组成：

- `welcome.html`
- `pages/...`
- `css/...`
- `js/...`
- `assets/...`

其中：

- 根目录 `welcome.html` 是壳页面。
- `pages/...` 是各个页面的内容源码。
- `css/...` 和 `js/...` 是样式与脚本源码。
- `welcome.html?page=xxx` 会通过动态加载脚本去请求对应的 `pages/...` 内容并渲染到主区域。

这意味着大多数页面在开发阶段，不一定要先构建才能看到修改结果。

### 浏览态

浏览态就是 `dist/...` 下的构建产物。

- `build.js` 会把 `welcome.html` 作为壳页面。
- 再把 `pages/...` 的内容注入进去，生成 `dist/*.html`。
- `npm run preview` 只会读取 `dist` 目录，不会直接读取源码目录。

因此：

- 如果你正在看 `dist/welcome.html` 或 `npm run preview` 打开的页面，源码改完后通常需要重新执行 `npm run build`。
- 如果你正在看根目录 `welcome.html` 对应的源码态页面，很多改动可以直接看到，不必每次都构建。

## 日常开发建议

推荐把日常流程固定成下面这样：

1. 改页面内容时，优先改 `pages/...`
2. 改全局导航、页头、页脚、壳结构时，改根目录 `welcome.html`
3. 改样式时，改 `css/...`
4. 改交互逻辑时，改 `js/...`
5. 联调源码效果时，打开根目录 `welcome.html`
6. 验证最终产物时，再执行 `npm run build` 查看 `dist`

## 如何打开开发态页面

不要直接用 `file://` 方式双击打开 HTML。

原因是根目录 `welcome.html` 里的动态加载逻辑会去请求 `pages/...` 文件，直接用本地文件协议打开时，通常会因为请求限制导致页面内容加载失败。

推荐方式：

- 用 Live Server 直接服务项目根目录
- 或使用任意静态服务器，以项目根目录 `dev` 作为站点根目录

开发时应优先打开：

- `/welcome.html`
- `/welcome.html?page=xxx`

而不是优先打开：

- `/dist/welcome.html`

## 解决方案页的特殊情况

解决方案页和大多数普通页面不一样，它们不是通过 `welcome.html?page=xxx` 动态加载的常规页面，而是由构建脚本结合数据生成的页面。

相关数据主要在：

- `data/solutions.js`

对应构建产物包括：

- `/dist/solution-retail.html`
- `/dist/solution-manufacturing.html`
- `/dist/solution-ecommerce.html`

因此这类页面的开发心智应该是：

- 改数据或模板后，执行 `npm run build`
- 再去 `dist` 里查看最终效果

## 什么时候必须 build

下面这些情况，通常都需要重新执行一次 `npm run build`：

- 你正在查看 `dist/...` 页面
- 你使用的是 `npm run preview`
- 你在验证最终发布产物
- 你修改的是解决方案页这类构建生成内容
- 你需要确认构建后的路径、资源引用、SEO 信息是否正确

## 什么时候通常不必 build

下面这些情况，通常可以先不构建，直接看源码态：

- 你修改了 `pages/...` 的普通页面内容
- 你修改了 `css/...` 样式
- 你修改了 `js/...` 的前端交互
- 你修改了根目录 `welcome.html` 的壳结构，并且当前就是通过源码态入口访问

## 推荐心智模型

可以把这个项目简单理解成下面两层：

- 第一层：源码层
  - `welcome.html`
  - `pages/...`
  - `css/...`
  - `js/...`
- 第二层：产物层
  - `dist/...`

日常开发优先盯源码层，发布验收再看产物层。

如果你发现“改了没生效”，先问自己两个问题：

1. 我现在看的到底是源码态页面，还是 `dist` 页面？
2. 我改的是源码文件，还是构建产物文件？

只要先把这两个问题分清，绝大多数“为什么没生效”的困惑都能很快定位。

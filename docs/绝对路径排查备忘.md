# 绝对路径排查备忘

## 背景

当前项目源码中存在较多以 `/` 开头的内部路径，例如 `/css/...`、`/js/...`、`/assets/...`、`/welcome.html...`、`/pages/...`。

这类写法在部署到站点根目录时通常可以正常工作，但如果静态站点发布到二级目录，例如 `/shimo/`、`/static/site/`，就容易出现以下问题：

- 图片加载失败
- 样式或脚本资源 404
- 页面跳转跳回站点根目录
- 运行时 `fetch` 子页面失败

本备忘仅记录源码层排查结果，不包含 `dist/` 目录内容。

---

## 总体结论

### 1. 当前项目确实大量使用内部绝对路径

问题主要集中在：

- `pages/` 下的模板页面
- `js/components/home-content-loader.js`
- 少量根入口页残留路径

### 2. 不是所有绝对路径都适合直接手改为相对路径

尤其是下面两类：

- `data-merge-source="/pages/..."`
- `home-content-loader.js` 中的 `source: "/pages/..."`

它们不仅是普通链接，还参与了运行时内容加载和构建期内联，不能简单做字符串替换。

### 3. 根入口页相对健康

- `welcome.html` 大部分资源与跳转已经是相对路径
- `home.html` 基本为相对路径
- 但 `welcome.html` 仍残留 1 处内部绝对路径

### 4. 构建脚本本身已经支持 base path

`build.js` 已支持通过 `--base` 将根路径改写为带前缀的路径，用于发布到二级目录，例如：

```bash
npm run build:shimo
```

对应：

```bash
node build.js --base=/shimo/
```

---

## 已确认的关键实现

### 构建脚本

`build.js` 中存在 base path 改写逻辑：

- `withBasePath(url)`
- `applyBasePathToHtml(htmlText)`
- `applyBasePathToJs(jsText)`

说明当前项目原本就是按“源码允许使用根路径，构建时再按部署前缀重写”的思路设计的。

### 文档说明

`docs/目录结构.md` 已明确写到：

- `data-merge-source` 统一使用绝对路径
- 构建时会按 `--base` 重写 `dist/` 内 HTML/JS 中的根路径资源引用

因此，当前问题更像是：

- 源码设计依赖 base path 构建
- 但实际打包、预览或部署时，没有完全匹配最终发布路径

## 风险分类

## 可以直接考虑改为相对路径的类型

这类一般可通过文件层级直接换算：

- `link href="/css/..."`
- `script src="/js/..."`
- `img src="/assets/..."`
- 普通页面跳转，例如 `/welcome.html...`

但需要按文件所在目录分别计算：

- 根目录页面
- `pages/features/`
- `pages/products/`
- `pages/legal/`
- `pages/solutions/`

## 不能简单全量替换的类型

这类不能只做字符串替换，必须结合构建逻辑或运行时解析一起处理：

- `data-merge-source="/pages/..."`
- `home-content-loader.js` 中的 `source: "/pages/..."`

原因：

- 它们参与运行时 `fetch`
- 它们同时参与构建时内联
- 它们的解析基准并不总是“当前源码文件目录”

---

## 优先级建议

如果后续要处理，建议优先看下面几块：

### 第一优先级

- `js/components/home-content-loader.js`
- `pages/home/home-content.html`
- `pages/products/office-suite-saas.html`
- `pages/products/office-suite-private.html`
- `pages/products/weboffice-sdk.html`
- `pages/features/collaboration.html`

原因：

- 这些文件涉及动态加载、主导航跳转、图片资源和核心产品页
- 对“打包后静态部署是否正常”影响最大

### 第二优先级

- `welcome.html`
- `pages/legal/*.html`
- `pages/solutions/*.html`

原因：

- 这些文件要么问题较少，要么主要是资源路径和跳回首页链接

---

## 当前建议

### 推荐做法

优先确认实际部署前缀，并使用已有构建能力进行验证：

```bash
npm run build:shimo
```

或：

```bash
node build.js --base=/你的发布目录/
```

如果部署前缀固定，这通常是成本最低的方案。

### 长期更稳妥的方向

如果后续需要适配多个不同部署目录，建议考虑增强构建逻辑，而不是直接全量手改源码：

- 保留源码中的“仓库根语义路径”
- 在构建输出阶段按目标文件位置改写成相对路径

这样风险会比“直接全量替换源码中的绝对路径”更低。

本次排查结论：

- 项目确实存在大量内部绝对路径

- 当前项目并非完全不支持子路径部署，而是更依赖构建时正确传入 `base path`
